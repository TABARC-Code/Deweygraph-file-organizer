# Archival Worthiness Filter

This is the first gate.

Before assigning anchors, links, or notes, decide whether an item deserves archival treatment at all.

A lot of systems skip this and end up lovingly cataloguing installer packages, duplicate screenshots, and PDFs nobody has opened since the coalition government. That is not sophistication. That is administrative cosplay.

---

## Purpose

Use this filter before deep classification.

It decides whether an item should be treated as:
- archival material
- reference material
- temporary intake material
- residue
- duplicate candidate
- discard candidate

---

## Primary questions

For each item, ask:

1. **Does this item have ongoing value?**
2. **Is it part of a meaningful project or corpus?**
3. **Would I want to find this again on purpose?**
4. **Does it contain original thought, process, evidence, or finished output?**
5. **Is it merely a transient file, download residue, or export duplicate?**

---

## Worthiness classes

### 1. Archival
Use when the item has clear long-term value.

Typical examples:
- core project notes
- design source files
- meaningful drafts
- technical notebooks
- original research material
- final outputs worth keeping
- project-defining references

Action:
- classify fully
- generate note if appropriate
- add links and decision note where needed

### 2. Reference
Use when the item matters, but mainly as support material.

Typical examples:
- external papers
- manuals
- inspiration images
- copied articles
- standards documents

Action:
- classify lightly
- avoid overlinking
- note generation optional

### 3. Process
Use when the item captures active making, iteration, or intermediate development.

Typical examples:
- work-in-progress sketches
- prototype photos
- lab notes
- iteration exports with traceable value

Action:
- classify if connected to a project corpus
- often link to parent collection

### 4. Temporary / Intake
Use when the item has not yet earned archival status.

Typical examples:
- downloads awaiting triage
- unreviewed exports
- unsorted scans
- installer packages

Action:
- do not deeply classify yet
- move to intake or triage zone first

### 5. Residue
Use when the item is noise left behind by ordinary computing.

Typical examples:
- cache artefacts
- redundant exports
- accidental duplicates
- empty placeholders
- screenshots with no enduring relevance

Action:
- do not classify
- archive or discard according to context

### 6. Duplicate Candidate
Use when the item appears to duplicate another item or version.

Action:
- do not treat as authoritative without further evidence
- hold for duplicate review

---

## Minimum threshold for full classification

An item should usually pass at least one of these tests before receiving a full record:

- it is original work
- it is part of a meaningful project corpus
- it has clear future retrieval value
- it is a significant reference
- it records process that would be useful later

If it does not pass those tests, it probably does not need a full note.

---

## Hard rule

**Do not dignify every file with metadata.**

Some files are just files.
Some are litter.
A decent system knows the difference.

---

## Use in pipeline

Apply this filter before:
- anchor assignment
- relation building
- note generation

This avoids overclassification and keeps the archive from turning into a museum of digital crumbs.
